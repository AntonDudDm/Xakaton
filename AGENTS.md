# AGENTS.md

## О проекте

Это хакатонный ML-проект по данным LMS Цифриум.

Цель проекта:
- провести EDA по поведенческим данным студентов;
- понять, какие паттерны связаны с оттоком;
- собрать целевой признак оттока;
- подготовить признаки;
- построить baseline-модель вероятности того, что студент не закончит курс.

Основная единица анализа:
**одна строка = один пользователь на одном курсе**.

Базовая таблица:
- `users_courses`

Все остальные таблицы должны по возможности агрегироваться до уровня `user-course`.

---

## Главное правило по target

Нельзя использовать `users_courses.state` как готовый таргет.

Target должен строиться отдельно, как комбинированный признак, на основе:
- активности на курсе,
- прекращения активности,
- незавершения курса.

Основные таблицы для target:
- `users_courses`
- `wk_users_courses_actions`
- `user_answers`

Если формула target пока не окончательна, это нормально.
В таком случае нужно:
1. предложить несколько candidate target definitions;
2. явно описать каждую;
3. не зашивать жёсткие assumptions без комментариев.

## Работа с кириллицей и кодировками

- Всегда читаем и записываем текстовые файлы через явный `encoding='utf-8'` (например, `Path(...).read_text(..., encoding='utf-8')`).
- Перед запуском вспомогательных скриптов и выводом отчётов выставляем `PYTHONIOENCODING='utf-8'` и, при необходимости, выполняем `chcp 65001`, чтобы консоль не превращала кириллицу в `???`.
- В ноутбуках и комментариях используем стандартные русские символы и избегаем спецсимволов вроде `→`, заменяя их на безопасные `->`.
- При генерации `.ipynb` через `nbformat` обязательно передаём `encoding='utf-8'` при чтении и записи.
- После автогенерации markdown или кода быстро проверяем предпросмотр (`nbformat`, `nbconvert`), что кириллица отображается корректно.

## Основные таблицы и как с ними работать

### `users_courses`
Главная таблица проекта.
Использовать как master table.

...### `wk_users_courses_actions`
Главный журнал действий пользователя на курсе.
Это основной источник для:
- анализа course activity;
- расчёта recency / frequency признаков;
- поиска последней активности;
- построения target.

### `user_answers`
Главный источник признаков по задачам:
- solved / skipped
- attempts
- points
- resource_type
- submitted_at

Использовать как источник task-level engagement признаков.

### `user_lessons`
Источник пользовательского прогресса по урокам.
Использовать для агрегации по `users_course_id`.

### `lessons`
Использовать в первую очередь как источник **course-level метаданных**.
Агрегировать по `course_id`, а не пытаться везде делать хрупкие row-level join'ы.

### `user_activity_histories`
Дополнительный источник.
Не использовать как основу всего пайплайна, если связь с `users_courses` не валидирована надёжно.

### Остальные таблицы
Использовать как обогащение:
- `users`
- `user_trainings`
- `user_access_histories`
- `user_award_badges`
- `award_badges`
- `wk_media_view_sessions`

## Что нужно сделать с EDA

Нужно пройтись по существующему `EDA.ipynb` и сделать **новый структурированный notebook**, сохранив текущую логику, но разделив ёё на понятные этапы.

Новая структура должна быть такой:

### Этап 1. Подготовка и чистка таблиц
В этот этап входит:
- чтение данных;
- очистка названий столбцов;
- удаление технических колонок;
- приведение id к корректному формату;
- приведение дат;
- обработка bool / category / numeric признаков;
- дроп очевидно лишних полей;
- фиксация проблем качества данных;
- проверка дубликатов;
- базовый schema audit.

### Этап 2. Слияние таблиц и подготовка новых фичей
В этот этап входит:
- агрегирование `lessons` по `course_id`;
- агрегирование `user_lessons` по `users_course_id`;
- агрегирование `wk_users_courses_actions` по `users_course_id`;
- аккуратная агрегация `user_answers`;
- агрегации по доступам, бейджам, пользователям и другим enrichment-таблицам;
- merge в итоговую таблицу уровня `user-course`;
- создание новых признаков;
- документирование логики merge и ограничений.

### Этап 3. Анализ подготовленных таблиц
В этот этап входит:
- описание итоговой таблицы;
- missing values;
- распределения ключевых признаков;
- анализ поведения пользователей;
- анализ активности / неактивности;
- анализ прогресса;
- анализ course-level признаков;
- формулирование гипотез для оттока;
- подготовка выводов, полезных для будущего target и модели.

# Хакатон — прогнозирование оттока студентов в LMS Цифриум

## О проекте

Этот репозиторий содержит решение кейса по анализу оттока студентов на платформе онлайн-обучения Цифриум.

Цель проекта:
- провести EDA по данным LMS;
- понять, какие поведенческие паттерны связаны с тем, что студент бросает курс;
- собрать целевой признак оттока;
- построить baseline-модель, которая предсказывает вероятность того, что студент не закончит курс.

## Постановка задачи

По брифу кейса требуется:
- проанализировать поведенческие данные студентов в LMS;
- выделить признаки, связанные с риском оттока;
- обучить модель прогнозирования оттока;
- описать ключевые поведенческие паттерны, связанные с бросанием курса.

Важно:
- отток нельзя брать напрямую из одного готового столбца;
- `users_courses.state` недостаточен для разметки, потому что студент может оставаться `active`, даже если фактически перестал учиться;
- целевой признак нужно собирать отдельно на основе активности, завершения курса и динамики поведения.

## Основная сущность анализа

Основная единица анализа в проекте:

**одна строка = один пользователь на одном курсе**

Базовая таблица:
- `users_courses`

Именно к этому уровню далее агрегируются остальные таблицы.

## Описание таблиц

### 1. `users_courses`
Главная таблица проекта.

Содержит информацию о прохождении курса пользователем:
- статус доступа;
- даты старта и завершения курса;
- баллы;
- максимальное число уроков, заданий и баллов по курсу.

Используется как базовая таблица для итогового датасета.

---

### 2. `wk_users_courses_actions`
Главный журнал действий пользователя на курсе.

Содержит:
- `users_course_id`
- тип действия (`visit_video`, `user_answer`, `start_training`, `visit_translation` и т.д.)
- время действия
- иногда `lesson_id`

Эта таблица особенно важна для:
- построения target;
- анализа активности / неактивности;
- расчёта признаков по частоте, давности и типам действий.

---

### 3. `user_answers`
Данные по ответам пользователя на задания.

Содержит:
- `user_id`
- `task_id`
- число попыток
- solved / skipped
- баллы
- тип активности (`Lesson`, `Homework`, `Training`)
- время отправки ответа

Используется для признаков, связанных с:
- вовлечённостью в задания;
- долей решённых задач;
- числом попыток;
- динамикой решения задач.

---

### 4. `user_lessons`
Информация о прохождении уроков пользователем внутри курса.

Содержит:
- `lesson_id`
- `users_course_id`
- просмотр видео / трансляции
- решённость урока
- количество решённых задач
- баллы по уроку

Используется для агрегации прогресса пользователя по курсу.

---

### 5. `lessons`
Описание уроков курса.

Содержит:
- `course_id`
- признаки наличия конспекта, задач, тренажёров;
- число задач;
- максимальные баллы;
- длительность видео;
- номер урока, если он есть.

Планируемое использование:
- агрегировать `lessons` по `course_id`;
- строить course-level признаки;
- затем присоединять их к `users_courses`.

---

### 6. `user_activity_histories`
Дополнительная история действий пользователя в LMS.

Содержит:
- `user_lesson_id`
- тип действия
- время действия

Таблица полезная, но связи с остальными таблицами могут быть грязными и не всегда однозначными. Поэтому она рассматривается как дополнительный источник, а не как основа пайплайна.

---

### 7. `user_trainings`
Информация о прохождении тренингов пользователем.

Можно использовать для признаков:
- число тренингов;
- попытки;
- набранные баллы;
- статус завершения;
- оценки.

---

### 8. `user_access_histories`
История выдачи доступа к курсам.

Можно использовать для признаков:
- длительность доступа;
- дата начала и окончания доступа;
- тип доступа.

---

### 9. `users`
Информация о пользователях.

Полезные поля:
- тип пользователя;
- пол;
- класс;
- количество логинов;
- регион / школа;
- подписка;
- признак учителя.

Для анализа студентов нужно будет исключать учителей.

---

### 10. `user_award_badges` и `award_badges`
Информация о наградах пользователя.

Можно использовать как прокси мотивации и вовлечённости:
- количество наград;
- типы наград;
- время первой награды.

---

### 11. `wk_media_view_sessions`
Информация о просмотрах медиа-контента.

Полезна для признаков, связанных с:
- глубиной просмотра;
- долей просмотренных сегментов;
- типом контента;
- live / vod / записи.

## Как предполагается собирать target

Целевой признак оттока будет собираться отдельно, а не браться из одного поля.

Базовая идея такая:

Студент считается ушедшим / не завершившим курс, если:
1. у него была осмысленная активность на курсе;
2. затем он перестал совершать действия на курсе;
3. курс не был завершён.

Для этого в target планируется использовать:
- `users_courses`
- `wk_users_courses_actions`
- `user_answers`

Возможные сигналы для target:
- нет действий после некоторой даты;
- нет активности до конца доступа;
- курс не завершён;
- низкий итоговый прогресс;
- отсутствие решений задач на поздних этапах курса.

Финальная формула target будет зафиксирована отдельно после EDA и проверки качества связей между таблицами.

## Что нужно сделать в проекте

1. Проверить качество данных и связи между таблицами.
2. Почистить типы, даты, id и дубликаты.
3. Построить агрегаты по таблицам до уровня `user-course`.
4. Провести EDA и выделить гипотезы об оттоке.
5. Собрать целевой признак.
6. Построить baseline-модель прогноза оттока.
7. Подготовить отчёт с ключевыми инсайтами.
